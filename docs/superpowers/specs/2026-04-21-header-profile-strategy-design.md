# Header Profile 策略统一设计

日期：2026-04-21

## 1. 背景

当前渠道编辑中的“UA 策略”和“请求头模板”存在明显割裂：

- `User-Agent` 被单独管理
- 其他请求头又通过另一套模板机制管理
- 预置、已选态、模板保存、内容编辑分散在不同区域
- 用户必须理解两套半重叠模型，才能完成一次“模拟某个客户端请求头”的配置

这直接导致以下问题：

- 选中预置后，状态反馈不清晰
- 长 `User-Agent` 被直接塞进列表，视觉噪音高
- “模板名称”和“模板内容”关系不清楚
- 用户无法直观看到“当前渠道最终会用哪一组完整请求头”

本设计目标不是继续修补 `UA` 局部交互，而是统一抽象，把 `User-Agent` 纳入完整请求头配置模型中。

## 2. 目标

把当前“UA 策略 + 请求头模板”重构为单一体系：

- 统一抽象为 `Header Profile`
- 一个 `Header Profile` 表示一组完整请求头快照
- `User-Agent` 只是其中一个字段，不再单独管理
- 渠道页只负责“选择哪些 Profile 参与策略”与“按什么模式切换”
- 所有完整请求头内容编辑都在 `Profile` 维度完成，不在渠道页做零散临时编辑

## 3. 非目标

本轮设计不包含：

- 与上游真实客户端自动同步版本号
- Profile 之间字段合并策略
- 多用户共享同一自定义 Profile 的团队协作能力
- 脱离渠道页的全局资产管理后台
- 改写现有请求头覆盖语义本身

本轮明确不做：

- 保留 `UA` 作为独立策略系统
- 保留“当前渠道里临时手填完整 Header，再另存为模板”的双状态设计

## 4. 核心抽象

### 4.1 Header Profile

`Header Profile` 是完整请求头快照，包含：

- `id`
- `name`
- `category`
- `scope`
- `headers`
- `readonly`
- `description`

建议语义：

- `name`
  - 人类可读名称，例如 `Chrome macOS`、`Codex CLI`
- `category`
  - 预置分组，例如：
    - `browser`
    - `ai_coding_cli`
    - `api_sdk`
    - `custom`
- `scope`
  - `builtin` 或 `user`
- `headers`
  - 完整 JSON 对象，例如：
  ```json
  {
    "User-Agent": "Mozilla/5.0 ...",
    "X-App": "codex-cli",
    "Anthropic-Beta": "..."
  }
  ```
- `readonly`
  - 预置 Profile 只读
  - 用户自定义 Profile 可编辑

### 4.2 渠道策略对象

渠道只保存策略选择结果，而不是再保存一份平行的请求头临时态。

建议策略对象包含：

- `enabled`
- `mode`
- `selected_profile_ids`

其中：

- `mode`
  - `fixed`
  - `round_robin`
  - `random`
- `selected_profile_ids`
  - 按顺序存储已选 Profile
  - 在 `fixed` 模式下只能有 1 个
  - 在 `round_robin` 模式下顺序有意义
  - 在 `random` 模式下仅表示候选池

## 5. 方案选型

本轮采用以下策略：

- 以 `完整请求头配置` 作为唯一主对象
- 允许多选多个完整 `Header Profile`
- 多选时仅支持：
  - `轮询`
  - `随机`
- 不做多个 Profile 字段合并
- 不保留“渠道页直接编辑当前完整 Header”的入口

原因：

- 心智模型最统一
- UI 最稳定
- 后续扩展浏览器 / CLI / SDK 预置最自然
- 不会再次出现“UA 是一套、模板又是另一套”的状态分裂

## 6. 信息架构

渠道编辑中的该区域统一重构为两层：

### 6.1 Header Profile 策略

上层为策略控制区，包含：

- `启用 Header Profile 策略`
- `策略模式`
  - 固定
  - 轮询
  - 随机
- `已选 Profile`

这个区域只负责：

- 当前是否启用
- 当前用什么模式
- 当前已选了哪些 Profile

不负责编辑完整请求头 JSON。

### 6.2 Profile 资源区

下层为可选资源区，包含：

- `预置 Profile`
  - 浏览器
  - AI Coding CLI
  - API SDK / 调试工具
- `我的 Profile`
  - 当前用户自己保存的自定义 Profile

该区域负责：

- 选择预置
- 选择自定义 Profile
- 新建 / 编辑 / 删除 / 复制自定义 Profile

## 7. 交互设计

### 7.1 渠道主流程

用户在渠道页中的操作路径为：

1. 打开 `Header Profile 策略`
2. 开启策略
3. 选择 `固定 / 轮询 / 随机`
4. 从资源区选择一个或多个 Profile
5. 保存渠道配置

### 7.2 固定模式

规则：

- 只能保留 1 个已选 Profile
- 用户选择新的 Profile 时，直接替换旧的已选项
- 不报错
- 给出即时提示：`已切换为新的固定 Profile`

### 7.3 轮询模式

规则：

- 允许多选
- 已选列表按顺序展示
- 支持拖拽排序
- 拖拽顺序即轮询顺序

### 7.4 随机模式

规则：

- 允许多选
- 不显示顺序编号
- 显示候选池数量

### 7.5 资源区选择行为

点击资源卡片时：

- 若未选中，则加入已选列表
- 若已选中，则从已选列表移除

状态必须即时反馈：

- 卡片选中态可见
- 按钮文案变化
- 已选列表同步更新

### 7.6 自定义 Profile

不在渠道页中直接输入完整请求头。

新增入口：

- `新建 Profile`

打开独立弹窗，字段包含：

- `名称`
- `分类`
- `Header JSON`
- `校验状态`
- `预览区`

保存后：

- 自动加入 `我的 Profile`
- 可以被渠道页直接选择

## 8. 视觉与动效规则

### 8.1 卡片状态

#### 默认态

- 复用当前系统已有卡片与边框语言
- 不引入重阴影、重玻璃效果或跳脱的视觉风格

#### Hover 态

- 边框强化
- 背景轻微提亮
- 垂直位移 `-1px`
- 出现辅助动作，例如：
  - `预览`
  - `添加`
  - `移除`

#### 选中态

- 必须不依赖 hover 才能识别
- 使用柔和底色 + 明确边框高亮
- 右上角可出现简洁选中标识

#### 禁用态

- 当策略关闭时，资源区弱化
- 但仍允许浏览和预览，不直接完全隐藏

### 8.2 已选列表

已选列表不再展示长字符串内容。

统一改为“名称行项目”：

- Profile 名称
- 分类标签
- 模式相关信息
- 右侧动作按钮

模式差异：

- `固定`
  - 强调“当前生效”
- `轮询`
  - 显示顺序编号和拖拽手柄
- `随机`
  - 仅显示已选数量，不显示顺序

### 8.3 动效

只使用轻量状态动效：

- 卡片 hover：`120ms`
- 选中切换：`160ms`
- 列表增删：`180ms`

原则：

- 动效只服务状态识别
- 不做多余后摇
- 不做拖沓切场

## 9. 预览设计

### 9.1 已选项显示规则

已选 Profile 只显示名称，不显示完整 Header 值。

### 9.2 悬停预览

鼠标悬停时显示预览浮层：

- 展示完整 Header JSON
- 支持滚动
- 支持复制
- `User-Agent` 字段应在视觉上优先可读

禁止：

- 把长 `User-Agent` 直接展示在 chip/tag 本体中
- 通过换行把一整段长 Header 撑高列表

## 10. 文案统一规则

为消除认知混乱，文案统一为：

- `User-Agent 策略` -> `Header Profile 策略`
- `UA 列表` -> `已选 Profile`
- `常用 UA 预置` -> `预置 Profile`
- `请求头模板` -> `我的 Profile`
- `保存为模板` -> `保存为 Profile`

必须移除所有继续暗示“单独管理 User-Agent”的旧文案，例如：

- `覆盖静态 User-Agent`
- `UA 策略模式`
- `UA 列表`

## 11. 自定义 Profile 表单规则

### 11.1 字段

- `名称`
  - 必填
  - 必须唯一
- `分类`
  - 默认 `custom`
- `Header JSON`
  - 必填
  - 必须为合法 JSON 对象

### 11.2 校验

- JSON 非法时：
  - 就地显示错误
  - 禁用保存按钮
- 名称重复时：
  - 就地提示 `名称已存在`
  - 提供：
    - 覆盖现有
    - 改名保存

### 11.3 空状态

当用户还没有自定义 Profile 时：

- 显示明确空状态说明
- 提供 `新建 Profile` 主入口
- 不出现大片空白

## 12. 数据与迁移建议

### 12.1 预置数据

预置 Profile 建议以前端静态注册表作为单一事实源维护。

建议字段：

- `id`
- `name`
- `category`
- `scope`
- `headers`
- `readonly`

### 12.2 旧配置迁移

旧系统中的以下配置需要迁移：

- 单独保存的 `User-Agent` 策略
- 旧的请求头模板引用

迁移原则：

- 若旧配置只包含 `User-Agent`
  - 自动包装为一个完整 `Header Profile`
- 若旧配置包含完整请求头
  - 尽量迁移为完整 Profile
- 不自动合并多个来源

迁移结果必须是：

- 渠道最终只引用 `selected_profile_ids`
- 不再同时持有另一套平行的 `UA` 状态

## 13. 错误反馈规则

### 13.1 选择行为

- 固定模式下再次选择新 Profile
  - 不报错
  - 直接替换
  - 给出轻提示

### 13.2 Profile 内容错误

- Header JSON 非法
  - 弹窗内就地报错
  - 禁止保存

### 13.3 空内容

- 明确提示：`Header JSON 不能为空`

### 13.4 无可选资源

- 显示空状态说明
- 引导到 `新建 Profile`

## 14. 验证策略

### 14.1 设计验证

需要重点验证以下体验目标：

- 用户是否还能感知成“两套功能”
- 已选状态是否无需 hover 即可识别
- 是否还能看到长值直接挤爆列表
- 自定义 Profile 的保存路径是否足够直接

### 14.2 前端交互验证

至少覆盖：

- 固定模式单选替换
- 轮询模式多选排序
- 随机模式多选候选池
- 悬停预览
- 新建自定义 Profile
- 编辑和删除 Profile
- 空状态
- 非法 JSON 报错
- 名称重复报错

### 14.3 视觉审查

必须同时检查：

- 亮色主题
- 暗色主题
- 桌面宽度
- 较窄窗口宽度

## 15. 残余风险

- 若后端当前仍以“单独 Header JSON”保存渠道配置，则实现阶段需要明确 `Profile 引用` 和 `最终 Header 快照` 的落库方式
- 若历史数据中存在大量旧 UA 配置，迁移策略需要做一次批量映射与回退预案
- 预置 Profile 的字段内容未来可能随上游客户端变化，需要后续维护，但这不影响当前抽象正确性

## 16. 最终结论

本轮不再继续优化“UA 小功能”，而是整体升级为统一的 `Header Profile` 体系。

关键决策如下：

- 以完整请求头配置为唯一主对象
- `User-Agent` 只是其中一个字段
- 渠道只选择 Profile，不直接零散编辑 Header
- 支持多选 Profile，并在它们之间执行：
  - 固定
  - 轮询
  - 随机
- 不做多 Profile 字段合并
- 不保留渠道页内临时 Header 编辑区

这是后续实现时唯一推荐方向。
