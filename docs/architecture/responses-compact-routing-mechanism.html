一句话先讲清楚

compact 后缀是内部路由模型

用户发 gpt-5.4 到 /v1/responses/compact，本地 new-api 会先把它变成 gpt-5.4-openai-compact 用于选渠道。

上游 body 里仍应放基础模型

native compact 请求转发给上游时，URL 是 /v1/responses/compact，body 的 model 通常是基础模型，例如 gpt-5.5。

跨模型必须显式映射

如果本地请求是 gpt-5.4-openai-compact，但上游只检测到 gpt-5.5-openai-compact，必须有 model_mapping 才允许走 5.5。

总流程

Step 1

Codex 客户端

发起 POST /v1/responses/compact，body 中模型通常是基础模型。

model = gpt-5.4

Step 2

本地 distributor

发现路径是 compact，就给模型名补内部后缀，用这个名字查可用渠道。

gpt-5.4-openai-compact

Step 3

渠道候选

先找精确 compact；找不到时才看基础模型能力，例如 gpt-5.4。

Step 4

native compact 门禁

只有 OpenAI 渠道开启 responses_compact_mode = native，并且 compact 能力有证据，才允许 fallback。

Step 5

模型映射

如果配置了显式 model_mapping，本地保留原始计费模型，但把上游 body 模型改成映射目标。

Step 6

上游 native compact

转发到上游 /v1/responses/compact。上游 new-api 会在自己的 distributor 中补 compact 后缀再选它自己的渠道。

为什么这样不会失忆

native 模式保留 compact 上下文

OpenAI adaptor 看到 native compact 后直接返回请求，不进入 convert 路径。

上游仍处理 compact endpoint

URL 仍是 /v1/responses/compact，上游响应是 response.compaction，包含 compaction_summary.encrypted_content。

改变的是上游模型，不是协议

本地日志和计费仍按原始 gpt-5.4-openai-compact，只是把上游 body 模型改成已知可 compact 的 gpt-5.5。

这是有意阻断。否则会把请求发给一个上游并不支持的 compact 模型，或者发生不可见的模型替换。

上游 /v1/models 不暴露 compact 后缀时

先分清两个事实

/v1/models 是发现信号

上游没有列出 *-openai-compact，只说明模型同步任务拿不到 compact 证据，不等于上游 compact endpoint 一定不可用。

/v1/responses/compact 是真实能力

如果上游本身也是 new-api，它收到 compact 路径后仍会在自己的 distributor 内部补 -openai-compact 后缀，再选它自己的渠道。

没有渠道就会 503

如果上游内部也找不到这个 suffixed model 的渠道，就会返回 No available channel for model ...-openai-compact。

本地 new-api 当前怎么处理
未检测过
如果上游模型检测关闭，或从未检测过，本地可以先按基础模型能力做 optimistic fallback。
已检测但无 compact
如果检测已执行且没有任何 compact 信号，本地默认不再自动 fallback，避免把请求发到明显没证据的渠道。
有显式映射
如果管理员配置了 model_mapping，本地会把它当作人工背书，允许转发给映射后的基础模型，让上游 compact endpoint 实测决定。
最终判定
真实准绳不是 /v1/models，而是 POST /v1/responses/compact 是否返回 response.compaction。

所以，这类上游最稳的接入方式是：先用真实 compact 请求探测；如果能 200，再用 model_mapping 表达“这个本地 compact 模型实际要发给哪个上游基础模型”。如果探测返回 503，就说明上游没有可用 compact 渠道。

两个模型名不要混淆

名字 当前例子 用途 为什么重要
OriginModelName gpt-5.4-openai-compact 本地计费、日志、用户看到的模型、重试上下文。 保证用户请求的模型语义可追踪，不把 5.4 请求静默记成 5.5。
UpstreamModelName gpt-5.5 实际写进转发给上游的 request body。 上游 new-api 在 compact 路径里会自己补后缀，最终选 gpt-5.5-openai-compact。
DetectedCompactModels gpt-5.5-openai-compact 渠道上次检测到的上游 compact 能力证据。 本地路由不再凭猜测选 native compact 渠道，减少 503 和失忆风险。

记忆口诀：本地选渠道看带后缀的内部模型；发给上游 body 用基础模型；跨模型只靠显式 model_mapping；native compact 不走 convert。
