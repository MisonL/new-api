# 邀请返利与礼品码说明

本文说明钱包/充值页中的邀请奖励、邀请记录、充值返利、邀请额度划转和礼品码能力。这里的“礼品码”不同于后台兑换码：礼品码由普通用户从自己的余额生成，用于定向或非定向赠送额度。

## 功能入口

- 用户入口：`/console/topup`
- 邀请码接口：`GET /api/user/aff`
- 邀请额度划转接口：`POST /api/user/aff_transfer`
- 邀请记录接口：`GET /api/user/invitation_records`
- 礼品码列表接口：`GET /api/user/gift_codes`
- 礼品码生成接口：`POST /api/user/gift_codes`
- 礼品码查看接口：`GET /api/user/gift_codes/:code`
- 礼品码接收接口：`POST /api/user/gift_codes/:code/receive`

## 管理员配置

邀请奖励相关配置位于系统设置的额度设置区域：

| 配置项             | 说明                                                                            |
| ------------------ | ------------------------------------------------------------------------------- |
| `QuotaForInviter`  | 新用户通过邀请码注册后，邀请人获得的邀请额度                                    |
| `QuotaForInvitee`  | 新用户使用邀请码注册后，被邀请人获得的额度                                      |
| `InviteRebateRate` | 被邀请用户充值成功后，按实际到账额度给邀请人发放的返利比例，合法范围为 0 到 100 |

`InviteRebateRate` 为 0 时，不发放充值返利。返利进入邀请人的 `aff_quota`，同时累加到 `aff_history`，不会直接进入账户余额；用户需要在钱包/充值页通过“划转到余额”转入余额。

## 邀请记录

邀请记录用于解释邀请收益从哪里来，当前支持两类记录：

| 类型     | `record_type`  | 说明                           |
| -------- | -------------- | ------------------------------ |
| 注册奖励 | `register`     | 被邀请用户注册后产生的邀请奖励 |
| 充值返利 | `topup_rebate` | 被邀请用户充值成功后产生的返利 |

筛选条件：

- `keyword`：搜索被邀请用户的用户名、邮箱或用户 ID。
- `record_type`：筛选注册奖励或充值返利。
- `source`：筛选记录来源。注册奖励来源为 `register`；充值返利来源通常是支付方式，例如 `stripe`、`creem`、`waffo` 或 Epay 具体支付方式。

展示方式：

- 列表视图：适合逐条核对收益、来源、返利比例和时间。
- 卡片视图：适合移动端或窄屏查看。
- 图表视图：按当前筛选条件聚合被邀请用户收益，帮助识别高价值被邀请人员。

为了兼容旧数据，邀请记录会把历史上只存在 `inviter_id` 关系但没有 `InvitationRecord` 行的注册关系补成只读展示记录；实际充值返利记录仍来自 `InvitationRecord` 表。

## 邀请充值返利

当被邀请用户完成充值后，后端按以下公式计算返利：

```text
返利额度 = 实际到账额度 * InviteRebateRate / 100
```

实际到账额度以最终入账给用户的额度为准，不以支付金额或原始展示金额为准。

当前已接入的充值完成路径：

- Epay 系列普通充值：`model.RechargeEpay`
- Stripe 普通充值：`model.Recharge`
- Creem 普通充值：`model.RechargeCreem`
- Waffo 普通充值：`model.RechargeWaffo`
- 管理员手动完成充值：`model.ManualCompleteTopUp`

每次成功发放返利后，会写入一条 `InvitationRecord`，并给邀请人记录系统日志。重复回调不会重复发放，因为充值订单完成逻辑会先检查订单状态。

## 邀请额度划转

邀请收益先进入 `aff_quota`，用户可在钱包/充值页打开“划转到余额”窗口，把邀请额度转入账户余额。

划转规则：

- 划转额度不能超过当前可用邀请额度。
- 划转额度必须达到系统最低划转单位。
- “全部划转”会自动填入当前全部可划转邀请额度。

## 礼品码

礼品码用于用户之间赠送余额。生成礼品码时会立即从生成用户余额扣除对应额度；只有接收成功后，额度才进入接收用户余额。

生成时可配置：

- 礼品额度：不能低于最低划转额度。
- 接收人绑定：可不绑定，也可按用户名、用户 ID 或邮箱绑定。
- 有效期：留空表示长期有效。
- 留言：接收人领取时可以看到。

生成成功后，页面会生成专用链接：

```text
/console/topup?gift_code=<code>
```

接收人未登录时会先进入登录流程。登录成功后，前端会保留原始 `gift_code` 查询参数并重新打开接收窗口。

接收时可填写感谢回信。接收成功后，系统会通知礼品码生成用户；通知内容会转义接收人名称和回信内容，避免用户输入被当成 HTML 渲染。

## 与兑换码的区别

| 能力   | 生成方   | 额度来源         | 使用方式                 |
| ------ | -------- | ---------------- | ------------------------ |
| 兑换码 | 管理员   | 后台兑换码配置   | 用户输入兑换码后领取     |
| 礼品码 | 普通用户 | 生成用户自身余额 | 接收人打开专用链接后领取 |

礼品码不替代后台兑换码。需要批量运营发码时仍使用兑换码；需要用户之间点对点赠送额度时使用礼品码。

## 排障口径

- 生成失败：先检查生成用户余额、礼品额度是否低于最低划转额度、绑定用户是否存在。
- 接收失败：先检查礼品码状态、有效期、是否已被领取、是否绑定了其他用户。
- 返利未到账：先检查 `InviteRebateRate` 是否为 0、被邀请用户是否存在 `inviter_id`、充值订单是否已经成功完成。
- 邀请记录不完整：先区分是旧注册关系补展示，还是充值返利记录没有写入 `InvitationRecord`。
- 礼品码链接登录后未弹窗：检查登录跳转是否保留了 `/console/topup?gift_code=...`。
